PROJECT: 5143 Xcentrics FTC Robot

SYSTEM OVERVIEW
===============

This implementation provides a production-ready autonomous system featuring:

1. Real-time Ball Detection
   - Husky Lens vision sensor integration
   - Multi-ball detection and prioritization
   - Distance-based ball ranking from frame center

2. Field Localization
   - AprilTag vision for pose fusion
   - Odometry integration with weighting (70% vision, 30% odometry)
   - Pose history for smooth estimates
   - Support for dual alliance colors (Red/Blue)

3. Autonomous Navigation
   - Pedro Pathing integration
   - Smooth ball navigation paths (Bezier curves)
   - Goal approach with turret pre-alignment

4. Turret-Based Scoring
   - Real-time turret aiming
   - Flywheel velocity control
   - Position tolerance: ±50 encoder ticks
   - Velocity tolerance: ±100 RPM
   - Multi-shot firing sequence

5. Driver Control & Feedback
   - Toggle button (gamepad1.Y): Enable/disable auto mode
   - Override button (gamepad1.X): Cancel autonomous at any time
   - Emergency stop (gamepad1.Back): Immediate halt of all systems
   - Haptic feedback: Vibration confirmation for mode changes
   - Rich telemetry: Real-time system status

6. State Machine Architecture
   - 10 well-defined states
   - Automatic timeout handling
   - Error recovery mechanism
   - Smooth mode transitions

TECHNICAL SPECIFICATIONS
========================

VISION INTEGRATION:

Ball Detection (Husky Lens):
├─ Frame: 320x240 pixels
├─ Algorithm: Object detection (configurable)
├─ Detection output: Position + size
├─ Update rate: ~20-30 FPS (non-blocking)
├─ Max balls tracked: Up to 20 per frame
└─ Filtering: Distance from center, size threshold

AprilTag Detection (Webcam):
├─ Processor: FTC AprilTagProcessor
├─ Tags supported: 36H11 family
├─ Pose estimation: Robot pose relative to tag
├─ Update rate: ~10-15 FPS per tag
├─ Fusion: Vision + odometry weighted average
└─ Validation: Distance-based update rejection

PATH FOLLOWING (Pedro Pathing):

Path Types:
├─ BezierLine: Direct ball navigation
├─ BezierCurve: Smooth goal approach
└─ Linear heading interpolation: Orientation control

Tolerances:
├─ Position: 2.0 inches
├─ Heading: Auto-interpolated
└─ Path completion: 90% of path (0.9 TValue)

TURRET CONTROL:

Position Control:
├─ Motor: DcMotorEx with encoder
├─ Encoder ticks per 360°: 4839.3
├─ Target tolerance: ±50 ticks
├─ Control: PID feedback loop
└─ Range: ±180° rotation

Velocity Control:
├─ Motor: Dual DcMotorEx (fly1, fly2)
├─ Velocity units: RPM (using setVelocity)
├─ Target velocity: 1300-1600 RPM
├─ Tolerance: ±100 RPM
└─ Control: Built-in motor velocity PID

Readiness Conditions (both must be met):
├─ Turret position error ≤ 50 ticks
└─ Flywheel velocity error ≤ 100 RPM


STATE MACHINE TIMEOUTS

Default Timeouts (seconds):
├─ SCAN: 3.0s (find ball)
├─ NAVIGATE_TO_BALL: 10.0s (reach ball)
├─ NAVIGATE_TO_GOAL: 10.0s (reach goal)
├─ ALIGN_TURRET: 3.0s (aim turret)
└─ WAIT_FOR_READY: 2.0s (ready status)

Customizable: All timeouts in AutoSystemConfig

DRIVER EXPERIENCE FLOW
======================

Scenario: Hybrid TeleOp Mode

1. Driver Startup (pre-match):
   - System initialized automatically
   - Status displayed on telemetry
   - Driver able to see current state

2. Normal TeleOp Play:
   - Driver controls robot manually
   - Gamepad feedback immediate

3. Decision: Enable Auto Assist
   - Driver presses Y
   - Gamepad vibrates (haptic feedback)
   - "AUTO MODE ENABLED" appears on telemetry
   - Turret/intake now autonomous
   - Drive still manual

4. During Auto Assist:
   - Robot finds and shoots balls automatically
   - Driver can still move robot with drive stick
   - Telemetry shows current state

5. Decision: Disable Auto Assist
   - Driver presses Y again
   - Gamepad vibrates (different pattern)
   - "AUTO MODE DISABLED" appears
   - Manual control restored

6. Emergency Situation:
   - Driver presses X (override)
   - Auto mode cancelled immediately
   - Or press Back (emergency stop)
   - All systems halt (motors powered off)
   - Haptic feedback confirms
